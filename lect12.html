Lesson 12: Inheritance, Polymorphism, and Encapsulation

Big picture

In this lesson we put together three major ideas of object-oriented programming: inheritance, polymorphism, and encapsulation. These ideas are related, but they answer different questions.

  • Inheritance asks: how are classes related?
  • Polymorphism asks: which method runs?
  • Encapsulation asks: who is allowed to touch the data?

Processing gives us a friendly place to experiment with these ideas. At the same time, Processing is sometimes more relaxed than standard Java, so we also need to point out what would happen in ordinary Java.

Recap: classes and objects

A class is like a blueprint or cookie cutter. It describes what data an object stores and what methods that object can perform. An object is an instance of the class.

Once we know how to build one object, a natural question is: can we reuse that design for a whole family of related objects? One important answer is inheritance.

Inheritance

Suppose we have a group of related things that share common features. We can put the common data and behavior into a more general class called a superclass (also called a parent class or base class). Then we can create more specific classes called subclasses that inherit from it.

Examples:

  • A trash-sorting machine may begin with a general class Trash, then create subclasses for cans, bottles, paper, and so on.
  • In geometry, a general shape class may lead to more specific shapes such as circles and squares.
  • On a farm, a general class Animal may lead to subclasses such as Cow, Pig, Duck, and Lamb.

Inheritance helps us avoid repeating the same code over and over.

The farm example

Here is a Processing example based on the idea of Old MacDonald’s farm. The class Animal stores what all animals in our sketch have in common: a position on the screen and a method animalSound(). Specific animals inherit from Animal and provide their own version of that method. 

class Animal {
  float x, y;

  Animal(float x, float y) {
    this.x = x;
    this.y = y;
  }

  void animalSound() {
    text("The animal makes a sound", x, y);
  }
}

class Cow extends Animal {
  Cow(float x, float y) {
    super(x, y);
  }

  void animalSound() {
    text("A moo moo here and a moo moo there", x, y);
  }
}

class Pig extends Animal {
  Pig(float x, float y) {
    super(x, y);
  }

  void animalSound() {
    text("An oink oink here and an oink oink there", x, y);
  }
}

class Duck extends Animal {
  Duck(float x, float y) {
    super(x, y);
  }

  void animalSound() {
    text("A quack quack here and a quack quack there", x, y);
  }
}

class Lamb extends Animal {
  Lamb(float x, float y) {
    super(x, y);
  }

  void animalSound() {
    text("A baa baa here and a baa baa there", x, y);
  }
}

In this example:

  • Animal is a superclass.
  • Cow, Pig, Duck, and Lamb are subclasses.
  • Each subclass inherits from Animal.
  • Each subclass overrides the method animalSound().

Overriding

When a subclass provides its own version of a method from the superclass, the subclass is said to override that method.

That is different from overloading, where methods have the same name but different parameter lists. In this lesson we are dealing with overriding.

For example, the Lamb class does not need to invent a new method name. It uses the same method name, animalSound(), but gives it more specific behavior.

Polymorphism

The Greek roots suggest “many forms.” In programming, polymorphism means that one reference type can refer to objects of different but related classes.

For example, this is legal: 

Animal myAnimal = new Animal(20, 40);
Lamb myLamb = new Lamb(20, 80);
Animal myPig = new Pig(20, 120);
Animal myCow = new Cow(20, 160);

Why is Animal myPig = new Pig(...) legal? Because a pig is an animal. The variable type is the more general superclass, while the actual object is a more specific subclass.

Then we can collect related objects into one array: 

Animal[] animals = {myAnimal, myLamb, myPig, myCow};

for (int i = 0; i < animals.length; i++) {
  animals[i].animalSound();
}

Each object responds in its own way. Even though the array type is Animal[], the correct version of animalSound() is chosen at runtime. That is polymorphism in action.

Why this matters

Inheritance helps us organize families of related classes. Polymorphism lets us treat many related objects in a uniform way. Instead of writing separate code for every animal, we can write a loop over an array of Animal references and trust each object to do the right thing.

Encapsulation

So far, inheritance and polymorphism help us organize and reuse code. Another major idea is encapsulation.

Encapsulation means that an object keeps its data and the methods that work with that data together, and it controls how that data can be changed.

This is important because without encapsulation, outside code can change object data in arbitrary ways, even ways that do not make sense.

Processing vs. Java

In Processing, we often skip strict access control for simplicity. That means beginner sketches frequently leave variables open and access them directly. This keeps examples short and readable.

In standard Java, however, access control is a central design tool. Fields are often declared private, which means they should be changed only by methods of the class itself. Other code interacts with the object through methods.

Processing is built on Java, but the PDE sketch environment is more relaxed. Because of that, private in a Processing sketch does not always behave as strictly as students should expect in ordinary Java. So Processing is excellent for learning ideas quickly, but it is not always a perfect model of professional Java design.

Access modifiers in Java

Java uses access modifiers to control visibility.

  • private: only this class can access the field or method
  • public: any code can access it
  • protected: subclasses can access it directly

In this lesson, the important contrast is between private fields and methods that form the public interface of the class.

Example 1: Account

The class below is a good example of encapsulation. The account stores its own data, and outside code is supposed to use methods such as deposit(), withdraw(), and getBalance(). 

class Account {
  private int accountNumber;
  private float accountBalance;

  Account(int number) {
    accountNumber = number;
    accountBalance = 50;
  }

  public int getAccountNumber() {
    return accountNumber;
  }

  public float getBalance() {
    return accountBalance;
  }

  public void deposit(float amount) {
    if (amount < 0) {
      println("Invalid deposit amount.");
    } else {
      accountBalance += amount;
    }
  }

  public void withdraw(float amount) {
    float fee = 1.25;
    if (amount < 0) {
      println("Invalid withdrawal amount.");
    } else if (amount + fee > accountBalance) {
      println("Insufficient funds.");
    } else {
      accountBalance -= (amount + fee);
    }
  }
}

This class does not provide a generic setBalance() method, and that is a good design choice. Encapsulation is not about writing getters and setters for everything. It is about deciding what operations make sense.

For a bank account, deposit() and withdraw() make sense. Randomly setting the balance does not.

Also note: in real financial software, using float for money is not a good idea because of rounding issues. We use it here only for simplicity.

Why the Account example is useful

Suppose anyone could directly write: 

acc.accountBalance = -9000;
acc.accountNumber = 5;

Then the methods deposit() and withdraw() would become almost meaningless, because outside code could ignore all the rules.

Encapsulation lets the object protect its own state. The object decides which changes are allowed.

It also encourages a good design question: should every piece of data be changeable? For example, an account balance changes over time, but an account number usually should not.

Example 2: Point

The Point example is especially good for CS1 because it is visual. We can see controlled change happen on the screen. 

Point p0, p1;

void setup() {
  size(500, 500);
  p0 = new Point(width/2, height/2);
  p1 = new Point(random(width), random(height));
}

void draw() {
  background(0);

  p0.display(color(255));
  p0.move(0, 1);   // controlled change through a method

  if (mousePressed) {
    p0.y = height/2;   // should be illegal outside the class in proper Java
  }

  p1.display(color(255, 0, 0));
  p1.setY(height + 20);   // setter prevents point from going below the window
}

class Point {
  private float x, y;

  Point(float x, float y) {
    this.x = x;
    this.y = y;
  }

  public float getX() {
    return x;
  }

  public float getY() {
    return y;
  }

  public void setX(float value) {
    x = constrain(value, 0, width);
  }

  public void setY(float value) {
    y = constrain(value, 0, height);
  }

  public void move(float dx, float dy) {
    setX(x + dx);
    setY(y + dy);
  }

  public void display(color c) {
    fill(c);
    ellipse(x, y, 10, 10);
  }
}

The important point is that the setters do something useful: they keep the point inside the window. The class is enforcing a rule, or invariant.

This is the main idea of encapsulation: the object does not merely store data; it protects that data by controlling how it changes.

Direct access vs. meaningful operations

Compare these two styles: 

p0.setY(p0.getY() + 1);   // controlled change
p0.move(0, 1);           // even better: meaningful operation

p0.y = height/2;         // direct access, bypasses the class

In good object-oriented design, we prefer meaningful methods such as move() over letting outside code manipulate raw data directly.

This is why encapsulation is more than “hide variables and add setters.” Objects should provide sensible operations and enforce sensible rules.

Common beginner misunderstanding

A common mistake is to think that encapsulation means: make everything private, then write getters and setters for everything.

That is not always good design. Sometimes a getter is fine. Sometimes a setter is not. Good encapsulation asks: what changes should this object allow?

Putting the three ideas together

These three ideas work together.

  • Inheritance gives us a family of related classes such as Animal, Cow, Pig, and Lamb.
  • Polymorphism lets us store them in one array of Animal references and call the same method name.
  • Encapsulation reminds us that each object should protect its own data and define sensible operations.

So object-oriented programming is not just “put variables and functions into a class.” It is also about designing relationships, behaviors, and boundaries.

Summary

  • Inheritance organizes related classes into superclasses and subclasses.
  • Overriding lets a subclass provide a more specific version of an inherited method.
  • Polymorphism lets one superclass reference type refer to many related subclass objects.
  • Encapsulation keeps data and behavior together and controls access to the data.
  • In Java, fields are often private and objects are manipulated through methods.
  • Processing is more relaxed, so it does not always enforce access control the way standard Java does.
  • Good design prefers meaningful operations such as animalSound(), deposit(), withdraw(), and move() over arbitrary direct changes to fields.
