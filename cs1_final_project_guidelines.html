CS 1: Final Project Guidelines

Read this carefully before you submit your proposal or start building.

1. What This Is

This is your final project—the big one. Your chance to take everything we’ve done this semester and turn it into something real.

Not just “code that runs,” but something that does something: a game, a visual, a tool, a weird interactive idea—your choice.

You are in charge here. That includes the fun parts and the frustrating parts.

2. Choose Your Path

Option A: Evolution

Take your midterm and make it significantly better.

Do not just tweak it—expand it.

In your proposal, be clear:

  • What exactly are you adding?
  • Why does it improve the project?
  • What new ideas are you using (arrays, classes, etc.)?

If it feels like the same project with small edits, it is not enough.

Option B: Genesis

Start from scratch.

If you have a new idea you’re excited about—go for it. This is often where the most interesting projects come from.

Game, animation, visualization, interactive story—anything is fine as long as it meaningfully uses what we learned.

3. What Must Be in Your Code

Your project should clearly use:

  • Variables
  • Conditionals and loops
  • Functions (to organize your code)
  • Arrays
  • Classes and objects

If one of these is missing, your project is not complete.

If you want to go further, great—consider:

  • Multiple screens (menu, gameplay, game over)
  • Motion or physics (velocity, gravity)
  • Images or pixel work
  • Text and typography
  • 3D (optional, not required)

4. What You Submit

Proposal

Answer these:

  • Project title
  • Evolution or Genesis
  • What your project actually does (2–3 sentences)
  • What programming structures you’ll use (be specific)
  • What you hope to add if things go well
  • What you think will be hard

Final Documentation (PDF)

Include:

  • Title and your name
  • Short summary
  • How to use your project (clear instructions)

Then reflect a bit:

  • How did you build it?
  • What went wrong at some point (something always does)?
  • How did you fix it?
  • What are you proud of?

Also include:

  • Credits (if you used anything external)
  • At least 3 screenshots:
  • The start screen
  • The main interaction
  • Another important state (game over, alternate mode, etc.)

5. Code Quality (This Matters)

  • Your code should be readable
  • Use reasonable names
  • Indent properly
  • Comment when something is not obvious

Your program should:

  • Run without crashing
  • Do what you claim it does
  • Work in fullscreen

Also:

  • Include a help screen (press i or similar)
  • Include a reset option (press r or similar)

If a user cannot figure out how to use it, that is a design problem.

6. AI Policy (Read This Carefully)

Do not use AI to generate your code or your write-up.

You can ask general questions and use AI as a search tool for exploration, but the actual work you submit must be yours.

Make sure you understand any code you submit—you may be asked to explain it.

If you use something we did not cover in class, be ready to explain it.

Final advice: Start early. Seriously. If you wait until the last minute, you will spend that time debugging instead of building something interesting. Getting stuck is normal. That is where learning happens.
