Lesson 12: Encapsulation

Big picture

So far, we have talked about classes, objects, inheritance, and polymorphism. Another major object-oriented programming idea is encapsulation.

Encapsulation means that an object keeps its data and the methods that work with that data together, and controls how that data can be changed.

In short:

  • Inheritance asks: how are classes related?
  • Polymorphism asks: which method runs?
  • Encapsulation asks: who is allowed to touch the data?

Processing vs. Java

In Processing, we often leave variables open because it keeps examples short and easy to read. That is convenient in a beginner class, but it is not the whole story.

In standard Java, fields are often declared private, which means they are supposed to be accessed only inside the class itself. Outside code should interact with the object through methods.

Processing is built on Java, but the PDE sketch environment is more relaxed. As a result, private in Processing sketches does not always behave as strictly as students should expect in ordinary Java programs. So Processing is great for learning ideas quickly, but not always a perfect model of professional Java design.

Access modifiers in Java

In Java, access modifiers help control visibility:

  • private: only this class can access the variable or method
  • public: any code can access it
  • protected: subclasses can access it directly

In this lesson, our main focus is on private fields and public methods.

A first example: Account

The class below is a reasonable example of encapsulation. The account stores its own data, and outside code is supposed to use methods such as deposit(), withdraw(), and getBalance(). 

class Account {
  private int accountNumber;
  private float accountBalance;

  Account(int number) {
    accountNumber = number;
    accountBalance = 50;
  }

  public int getAccountNumber() {
    return accountNumber;
  }

  public float getBalance() {
    return accountBalance;
  }

  public void deposit(float amount) {
    if (amount < 0) {
      println("Invalid deposit amount.");
    } else {
      accountBalance += amount;
    }
  }

  public void withdraw(float amount) {
    float fee = 1.25;
    if (amount < 0) {
      println("Invalid withdrawal amount.");
    } else if (amount + fee > accountBalance) {
      println("Insufficient funds.");
    } else {
      accountBalance -= (amount + fee);
    }
  }
}

Notice that this class does not provide a generic setBalance() method. That is a good design choice. Encapsulation is not about writing getters and setters for everything. It is about deciding what operations make sense.

For a bank account, deposit() and withdraw() make sense. Randomly setting the balance does not.

Also note: in real financial software, using float for money is not a good idea because of rounding issues. We use it here only for simplicity.

Why encapsulation matters

Suppose anybody could directly write: 

acc.accountBalance = -9000;
acc.accountNumber = 5;

Then the methods deposit() and withdraw() would become almost meaningless, because outside code could ignore all the rules.

Encapsulation lets the object protect its own state. The object decides what changes are allowed.

A cleaner visual example: Point

The Point example is even better for CS1 because it is visual. We can actually see what happens when we change coordinates in a controlled way. 

Point p0, p1;

void setup() {
  size(500, 500);
  p0 = new Point(width/2, height/2);
  p1 = new Point(random(width), random(height));
}

void draw() {
  background(0);

  p0.display(color(255));
  p0.move(0, 1);   // controlled change through a method

  if (mousePressed) {
    p0.y = height/2;   // should be illegal outside the class in proper Java
  }

  p1.display(color(255, 0, 0));
  p1.setY(height + 20);   // setter prevents point from going below the window
}

class Point {
  private float x, y;

  Point(float x, float y) {
    this.x = x;
    this.y = y;
  }

  public float getX() {
    return x;
  }

  public float getY() {
    return y;
  }

  public void setX(float value) {
    x = constrain(value, 0, width);
  }

  public void setY(float value) {
    y = constrain(value, 0, height);
  }

  public void move(float dx, float dy) {
    setX(x + dx);
    setY(y + dy);
  }

  public void display(color c) {
    fill(c);
    ellipse(x, y, 10, 10);
  }
}

Here the setters do something useful: they keep the point on the screen. That means the class is enforcing a rule, or invariant.

This is the main idea of encapsulation: the object does not merely store data; it protects that data by controlling how it changes.

Important lesson from the Point example

Compare these two approaches: 

p0.setY(p0.getY() + 1);   // controlled change
p0.move(0, 1);           // even better: meaningful operation

p0.y = height/2;         // direct access, bypasses the class

In good object-oriented design, we prefer meaningful methods such as move() over letting outside code manipulate raw data directly.

This is why encapsulation is more than "hide variables and add setters." It means that objects provide sensible operations and enforce sensible rules.

Common beginner misunderstanding

A common mistake is to think that encapsulation means: make everything private, then write getters and setters for everything.

That is not always good design. Sometimes a getter is fine. Sometimes a setter is not. For example, an account number probably should not change after an account is created.

Good encapsulation asks: what changes should this object allow?

Summary

  • Encapsulation means keeping data and behavior together and controlling access to the data.
  • In Java, fields are often private and objects are manipulated through methods.
  • Processing is more relaxed, so it does not always enforce access control the way standard Java does.
  • Encapsulation is useful because methods can enforce rules and protect the object's state.
  • Good design prefers meaningful operations such as deposit(), withdraw(), and move() over arbitrary direct changes to fields.
